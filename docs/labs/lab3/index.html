Лабораторная работа №3

Настройка CI/CD для статического сайта с использованием SourceCraft

Бондарева Ирина | 2026

Цель работы

Организовать процесс непрерывной интеграции и развертывания (CI/CD) для статического веб-сайта, сгенерированного с помощью MkDocs, и подключить удаленный репозиторий на платформе SourceCraft.

Выполненные задачи:

  • Создание и настройка репозитория на SourceCraft
  • Генерация персонального токена для доступа к репозиторию
  • Добавление удаленного репозитория в локальный проект
  • Настройка автоматической сборки и публикации через конфигурационные файлы
  • Сравнение подходов CI/CD на GitHub и SourceCraft

Ход выполнения работы

1. Подготовка репозитория на SourceCraft

На платформе SourceCraft.dev был создан новый пустой репозиторий. Для обеспечения возможности удаленной работы с репозиторием был сгенерирован персональный токен доступа (Personal Access Token), который в дальнейшем использовался для аутентификации.

Подключение удаленного репозитория

В локальном проекте была выполнена команда для привязки удаленного репозитория SourceCraft:

git remote add sourcecraft https://<username>:<token>@git.sourcecraft.dev/<organization>/<repository>.git

Проверка корректности подключения осуществлялась командой:

git remote -v
# Результат выполнения:
origin      https://github.com/username/repo.git (fetch)
origin      https://github.com/username/repo.git (push)
sourcecraft https://git.sourcecraft.dev/org/repo.git (fetch)
sourcecraft https://git.sourcecraft.dev/org/repo.git (push)

2. Организация CI/CD процесса

Для автоматизации сборки и публикации сайта в корневой директории проекта были размещены конфигурационные файлы в папках .sourcecraft/ и .github/. Разработанный сценарий активируется автоматически после каждого выполнения git push в ветку main.

Конфигурация Cubes для SourceCraft

Этап сборки: используется Docker-образ python:3.14-slim, устанавливаются зависимости из файла requirements.txt, после чего выполняется генерация статических HTML-страниц командой mkdocs build.

Этап публикации: создается или переключается на ветку release, содержимое собранного сайта из директории ./site копируется в корень, фиксируется и отправляется в удаленный репозиторий.

# Пример конфигурации .sourcecraft/cubes.yaml
version: 1
cubes:
  - name: build-and-publish
    steps:
      - name: install-deps
        image: python:3.14-slim
        commands:
          - pip install -r requirements.txt
      - name: generate-site
        image: python:3.14-slim
        commands:
          - mkdocs build
      - name: publish-release
        image: alpine/git
        commands:
          - git checkout --orphan release || git checkout release
          - rm -rf .
          - cp -r ../site/* .
          - git add .
          - git commit -m "Deploy site"
          - git push origin release --force

Файл sites.yaml для привязки к хостингу

site:
  root: "site"
  ref: "main"

Данный файл указывает платформе SourceCraft, из какой ветки и из какой директории необходимо публиковать сайт.

Сравнительный анализ CI/CD подходов

GitHub Actions

  • Применяется встроенная утилита mkdocs gh-deploy
  • Утилита самостоятельно формирует временную директорию, инициализирует Git, копирует собранные файлы, создает коммит в ветке gh-pages и выполняет push
  • Доступны готовые действия (actions), оптимизированные для экосистемы GitHub

SourceCraft Cubes

  • Требуется ручная настройка всех этапов через кубы
  • Необходимо самостоятельно прописывать команды для создания ветки release, копирования файлов из папки site, выполнения коммита и push
  • Отдельный файл sites.yaml связывает выбранную ветку с веб-хостингом платформы

Результаты развертывания

После выполнения команды git push sourcecraft main на платформе SourceCraft была успешно запущена сборка сайта. В результате работы CI/CD все итоговые файлы сайта были размещены в ветке release репозитория, а сам сайт стал доступен по адресу, сгенерированному платформой.

Необходимые настройки для развертывания

GitHub Pages

В разделе Settings → Pages выбрать источник "Deploy from branch" и указать ветку gh-pages

SourceCraft

Создать публичную организацию, сгенерировать PAT с правами Maintainer, настроить cubes.yaml и sites.yaml

Локальный репозиторий

Добавить удаленный репозиторий командой git remote add sourcecraft

Дополнительные настройки

Для корректной работы GitHub Actions в репозитории GitHub необходимо в разделе Settings → Actions → General разрешить выполнение workflows. Для SourceCraft важно, чтобы репозиторий и организация были публичными — это обязательное условие для публикации сайта.

  • Создан и настроен удаленный репозиторий на платформе SourceCraft
  • Сгенерирован персональный токен доступа для аутентификации
  • В локальный проект добавлен второй удаленный репозиторий (sourcecraft)
  • Разработаны конфигурационные файлы для автоматической сборки и публикации
  • Выявлены ключевые различия в подходах CI/CD между GitHub Actions и SourceCraft Cubes
  • Сайт успешно развернут и доступен по адресам обеих платформ

Полученные навыки позволяют в дальнейшем организовывать автоматизированное развертывание статических сайтов с использованием различных CI/CD инструментов и платформ хостинга.
